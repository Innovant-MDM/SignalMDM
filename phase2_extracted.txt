1. Executive Summary 
• Phase 2 establishes the mapping and standardization layer for SignalMDM. 
• It consumes staging records created in Phase 1 with staging_status = READY_FOR_MAPPING. 
• It allows data architects and data engineers to define canonical entity models, configure source-to-canonical field mappings, 
apply transformation rules, apply standardization rules and produce normalized records. 
• The output of Phase 2 is not only a set of UI screens. It is a controlled canonicalization engine that converts inconsistent 
source data into master-data-ready records. 
• Phase 3 Data Quality will consume Phase 2 records with staging_status = READY_FOR_DQ. 
 
Phase 2 Output: Normalized canonical payloads stored against staging records, normalization run summaries, and 
mapping/normalization errors captured for remediation. Final eligible status: READY_FOR_DQ. 
 
2. Phase 2 Scope and Boundaries 
In Scope Out of Scope for Phase 2 
Canonical model management Data Quality rule scoring and DQ dashboards 
Source-to-canonical field mapping Matching, entity resolution and clustering 
Transformation rule configuration and execution Golden record creation and survivorship 
Standardization rule configuration and execution Stewardship workflows and approvals 
Normalization run execution and tracking External enrichment and signal intelligence 
Normalized record monitoring Downstream publishing or SignalIQ integration 
Mapping and normalization error handling Manual DQ remediation workflows 
Seed data for canonical fields, rules and reference values Production-grade data lake or raw document processing 
 
3. Implementation Principles 
• Canonical-first design: every source field must map into a controlled canonical field before later phases can trust it. 
• Metadata-driven mapping: mappings, transformations and standardization rules must be database-driven and configurable 
through UI. 
• Backend is the source of truth: frontend validation improves experience, but all business validation must exist in backend 
services. 
• No raw data mutation: raw_records and raw payload snapshots from Phase 1 must remain unchanged. 
• Idempotent normalization: re-running normalization for the same ingestion run must not create duplicate normalized outputs. 
• Audit everything: create, update, deactivate and retry actions must be captured in audit logs. 
• Clear statuses: staging entities must move through controlled statuses only. 
• Separation of concerns: do not mix DQ validation, matching or golden record logic into Phase 2.

6. User Flow 
1. Data architect opens the Canonical Models screen and defines canonical fields for entity types such as CUSTOMER, 
SUPPLIER, PRODUCT and ACCOUNT. 
2. Data engineer opens Field Mappings and selects source system + entity type. 
3. System displays source fields and sample values from staging records. 
4. Data engineer maps source fields to canonical fields. 
5. Data engineer assigns transformation rules where required. 
6. Data architect configures standardization rules for values such as country, state, phone, date and codes. 
7. User triggers a normalization run for an ingestion run / source system / entity type. 
8. Backend creates normalization_run and processes eligible staging records asynchronously. 
9. System stores canonical payloads and updates staging records. 
10. User monitors Normalization Runs and reviews normalized records or mapping errors. 
11. Successful records are marked READY_FOR_DQ for Phase 3.

16. Backend Service Logic 
Service Function Logic 
create_canonical_field(payload, user_context) Validate entity_type, snake_case field name, uniqueness, enum values; insert canonical_fields; write 
audit event. 
create_field_mapping(payload, user_context) Validate source system, canonical field, duplicate mapping and transformation rules; insert mapping; 
write audit event. 
run_normalization(payload, user_context) Create normalization_run; fetch eligible staging_entities; set status RUNNING; enqueue 
NORMALIZATION_JOB. 
process_record(staging_entity) Read raw_payload_snapshot_json; apply mappings; apply transformation chain; apply 
standardization; update payloads/status. 
handle_mapping_error(record, error) Persist mapping_errors row; update staging_entities.normalization_status and staging_status to 
NORMALIZATION_FAILED.

    run = get_normalization_run(normalization_run_id) 
    records = get_staging_records(

    update_run_status(run.id, 'RUNNING') 
 
    for record in records:

            update_staging_status(record.id, 'MAPPING_IN_PROGRESS') 
 
            mappings = get_active_mappings( 
                source_system_id=record.source_system_id, 
                entity_type=record.entity_type 
            ) 
 
            mapped_payload = {} 
            for mapping in mappings: 
                value = extract_value(record.raw_payload_snapshot_json, mapping.source_field_name) 
                if value is None: 
                    value = mapping.default_value

                value = apply_standardization(value, mapping.canonical_field_name) 
                mapped_payload[mapping.canonical_field_name] = value 
 
            update_staging_entity( 
                record.id, 
                mapped_payload_json=mapped_payload, 
                standardized_payload_json=mapped_payload,

records must exist, active mappings must exist. 
Mapping Errors Retry error must be OPEN, related staging record must exist, mapping/rule correction must be available before retry.
